# Quality Evaluator — 산출물 품질 평가 (project 스킬)

산출물을 검증하고 PASS/FAIL 판정을 내린다. FAIL 시 구체적 수정 지시를 제공하여 자동 수정 루프를 지원한다.

## 검증 3단계

### Layer 1: 파일 유효성 (결정론적)

**HWPX 문서**: 파일 크기 > 1KB, ZIP 구조 정상, mimetype 정합, `Contents/section0.xml`/`header.xml` 존재, 재오픈 검증(단락 수·표 구조), 스타일·표·머리글/바닥글 확인.

**DOCX 문서**: 파일 크기 > 5KB, ZIP 구조 정상, `word/document.xml`/`[Content_Types].xml` 존재, 재오픈 검증, 스타일(Heading·볼드·이탤릭)·표·페이지 설정·머리글/바닥글 확인.

**PPTX 문서**: 파일 크기 > 10KB, ZIP 구조 정상, `ppt/presentation.xml` 존재, 슬라이드 수 ≥ 요청 수, 제목 shape·폰트 크기·색상 일관성·레이아웃·이미지 placeholder 미잔존·마스터/테마 확인.

**XLSX 문서**: 파일 크기 > 5KB, ZIP 구조 정상, `xl/worksheets/sheet1.xml` 존재, 헤더 행 서식·열 너비·숫자 포맷·테두리·수식·시트명 확인.

**텍스트/마크다운**: 내용 길이 > 100자, 빈 파일 아님. **CSV/데이터**: 분석 결과 테이블 + 수치 계산 포함. **차트/시각화**: Mermaid 코드 블록 또는 HTML 파일 + 데이터 실제 반영.

### Layer 1.5: 마크다운 렌더링 버그 검수

문서 내 마크다운 문법(`**볼드**`, `*이탤릭*`, `` `코드` ``, `[링크](URL)`, `# 제목`)이 실제 문서(HWPX/DOCX/PPTX)에서 렌더링되지 않고 원본 그대로 노출되는 버그를 검출한다. 검출 시 마크다운 기호를 제거하고 라이브러리의 스타일 API(bold=True 등)를 직접 적용하도록 수정 지시한다.

### Layer 1.6: AI 작문 패턴 검수

산출물 텍스트에서 AI가 생성한 티가 나는 표현을 검출한다.

**AI 슬롭 단어 목록 (한국어, 예시)**: 혁신적인→새로운, 패러다임→방식/체계, 획기적인→눈에 띄는, 선도적인→앞서가는, 차세대→다음 단계의, 최첨단→최신, 완벽한 가이드→핵심 정리, 궁극적으로→결국, 다양한→구체적 숫자로 대체, 효과적인/효율적인→구체적 결과로 대체, 확실히→분명, 전반적으로→대체로, "~의 잠재력을 열어보세요"/"~의 여정"/"~의 세계로 빠져보세요"/"~에 대해 알아보겠습니다"→삭제.

**구조적 패턴**: "~합니다" 어미 3회 이상 연속 반복, 모든 문장이 비슷한 길이(리듬 단조로움), 느낌표 과도 사용(문서당 최대 1개), "~해보세요" 3회 이상 반복, 구체적 숫자/사실 없이 추상적 찬사만.

적용 범위: 모든 텍스트 산출물(보고서·제안서·카피·블로그 등), HWPX/DOCX/PPTX 내부 텍스트, 마크다운 산출물.

### Layer 2: 내용 완전성 (AI 판단)

사용자 요청과 산출물을 대조하여 누락 없는지 확인한다: 정확성(사실 오류·수치/날짜·법규 적합성), 근거(출처)(외부 사실 주장이 출처를 갖추거나 「확인 필요」 표시), 완전성(요청 항목 전수 포함), 실용성(즉시 사용 가능), 톤 적합성, 도메인 적합성, 마크다운 렌더링, AI 작문 회피.

### Layer 2.5: 근거 검증 (grounding)

내용이 AI 같지 않은가(Layer 1.6)와 별개로, 주장이 사실로 뒷받침되는가를 검증한다. Layer 1.6이 문체(AI 티)를 보는 반면, 이 단계는 진실성(주장이 근거되는가)을 본다.

**근거 강도 분류**:

| 강도 | 정의 | 판정 |
|------|------|------|
| **직접(grounded)** | 검증 가능한 출처(URL·문서명·공식 사이트·법령 조문)가 명시됨 | PASS |
| **일반(generic)** | "일반적으로", "대부분" 등 출처 없는 포괄 진술 | 약함 — 출처 보강 권장 |
| **미확보(ungrounded)** | 출처 없는 구체 수치·날짜·통계를 사실처럼 단정 | **FAIL** |

**검증 대상**: 수치(시장 규모·점유율·성장률·가격·통계), 날짜·기한(공고 마감·시행일), 고유 사실(법령·규정 조문명·정부지원 자격요건·경쟁사 동향·인용). 출처 없는 구체 수치·날짜·규정을 사실로 제시하면 FAIL(출처 명시 또는 「확인 필요 — 출처 미확보」 표시 요구). 변동 정보는 "최신 공식 자료 확인 필요"를 함께 표기. 창작·예시 목적의 가상 수치는 "예시"·"가정" 명시 시 통과. 검증 깊이는 `execution-protocol.md` §6-4 위험 등급(QUICK/NORMAL/DEEP)을 따른다.

### Layer 3: 수정 지시 생성

FAIL 시 구체적으로 무엇을 어떻게 수정해야 하는지 지시한다.

## 출력 형식

```
## 품질 평가 결과

**판정**: PASS / FAIL

### Layer 1: 파일 유효성
| 항목 | 결과 |
|------|------|
| 파일 존재 | OK/NG |
| 구조 정상 | OK/NG |
| 필수 요소 | OK/NG |

### Layer 2: 내용 완전성
| 차원 | 결과 | 비고 |
|------|------|------|
| 정확성 | OK/NG | ... |
| 근거(출처) | OK/NG | 미출처 수치/날짜/규정 여부 |
| 완전성 | OK/NG | ... |
| 실용성 | OK/NG | ... |
| 톤 적합성 | OK/NG | ... |
| 도메인 적합성 | OK/NG | ... |

### 수정 지시 (FAIL 시)
1. [구체적 수정 내용 — 파일/위치/변경사항]
```

## 파일 유효성 검증 명령어

```bash
unzip -t output.hwpx 2>&1 | tail -1
unzip -t output.docx 2>&1 | tail -1
unzip -t output.pptx 2>&1 | tail -1
unzip -t output.xlsx 2>&1 | tail -1
```

## 평가 원칙

- **회의적 기본 태도**: 의심스러우면 FAIL
- **출처 없는 단정 금지**: 검증 가능한 출처 없이 구체 수치·날짜·규정을 사실로 제시하면 FAIL
- **구체적 근거 필수**: PASS/FAIL 모두 이유 명시
- **수정 가능한 피드백**: FAIL 시 "이렇게 고쳐라"까지 제시
- **자기 평가 함정 방지**: 생성자의 자기 칭찬 무시
- **한국어 응답**

## 반복 개선 루프

```
생성 → quality-evaluator 평가
         ↓
    PASS → 사용자에게 전달
    FAIL → 수정 지시 반환 → 재생성 → 재평가(최대 3회)
         ↓
    3회 후에도 FAIL → 현재 최선본 + 문제점 사용자에게 보고
```
