# story-series-bible: 연재 상태 원장

장기 연재의 **상태를 기억하는 원장**이다. 마스터 기획서 파일 하나를 소유하고, 회차를 쓸 때마다 그 안의 **에피소드 현황표**를 갱신한다. 웹툰·웹소설 두 라인에 같은 메커니즘으로 쓰인다. 체인의 시작(상태 확립)과 끝(상태 갱신)에 두 번 등장하는 루프의 축이다.

## 1. 왜 원장이 필요한가

장기 연재는 세션이 수십 번 끊긴다. 매번 백지에서 시작하면 5화쯤부터 캐릭터·복선·톤이 표류한다. 원장은 **이전 회차까지의 상태를 복원**해 이 표류를 막는다.

- 기획은 산출하고 끝나는 게 아니라 **회차마다 갱신되는 살아있는 문서**다.
- 이 스킬은 스토리 창작(story-webtoon-planner·webnovel-writer)과 별개로 **상태 관리**만 전담한다.

## 2. 두 종류의 파일

이 스킬은 두 종류의 파일을 만든다. 파일명 규약은 `references/filename-convention.md`.

| 산출물 | 성격 | 파일명 | 생성 시점 |
|--------|------|--------|-----------|
| **전체 기획서** | 시리즈 설계도(세계관·캐릭터·아크) | `[작품제목]_기획서.md` | 신규 기획 시 **1회 생성**, 이후 현황표만 갱신 |
| **에피소드 원고** | 그 화의 완성본(+비주얼) | `[작품제목]_EP##_[에피소드제목].md` | 각 화 작성 때마다 |

## 3. 2-모드 진입 (가장 먼저 실행)

기획서 파일 유무로 모드를 결정한다.

### 케이스 A — 기획서 있음 (시리즈 재호출)

사용자가 기존 `[작품제목]_기획서.md`를 제시하거나 "기획서 있어, ep## 만들어줘"라고 하면:

1. **기획서 전체를 읽는다.** 요약만 보지 않는다.
2. **큰 그림을 파악한다:**
   - 지금까지 몇 화가 나왔는가
   - 독자가 현재 어떤 감정 상태일 것인가 (각 화 흐름으로 추정)
   - 전체 아크에서 이번 화가 해야 할 역할은 무엇인가
   - 시리즈 목적(인기작)을 위해 이번 화가 해줘야 하는 것은
3. 그 판단을 기반으로 회차 작성으로 바로 진행한다.

> ⚠️ 기획서의 각 화 요약만 꺼내 쓰지 않는다. **전체 흐름에서 역산해 이번 화를 설계**하는 것이 이 스킬의 존재 이유다. 역산 5원칙은 `references/recall-design.md`.

### 케이스 B — 기획서 없음 (신규 기획)

마스터 기획서를 새로 설계한다. 골격은 `references/bible-template.md`.

1. 독자 페르소나 정의
2. 세계관 설계
3. 캐릭터 설계 (서사 정보 + 비주얼 시트 + 일관성 앵커)
4. 시리즈 전체 아크 + 훅 전략
5. **전체 기획서 저장** — 맨 아래에 에피소드 현황표를 넣어 둔다.

## 4. 불변식 — 에피소드 생성 = 현황표 갱신

**회차 원고를 만들면 반드시 기획서의 에피소드 현황표를 갱신한다. 항상 세트다.**

- 갱신하지 않으면 다음 재호출 시 상태 복원이 무너진다.
- 현황표 스키마: 화 / 상태 / 제목 / 핵심 감정 / 훅 포인트 / 파일 / 날짜. (`references/bible-template.md`)

```
| ep## | ✅ 작성됨 | [에피소드제목] | [핵심 감정] | [훅 포인트 요약] | [파일명] | [날짜] |
```

## 5. 워크플로우 요약

```
Step 0. 모드 확인 (기획서 유무)
Step 1. [신규] 마스터 기획서 설계 → 저장 + 현황표 삽입
        [재호출] 기획서 전체 읽기 → 역산으로 이번 화 역할 판단
Step 2. 회차 원고 작성 (창작은 planner/webnovel-writer 체인에 위임 가능)
Step 3. 현황표 갱신 (불변식 — 반드시)
```

## 6. 워크드 예시

### 케이스 A — "기획서 있어, ep05 만들어줘"

1. `우리집에는_철학자가_산다_기획서.md` **전체를 읽는다**(요약만 X).
2. 현황표에서 ep01~04가 ✅임을 확인, 각 화 핵심 감정·훅으로 **독자의 현재 감정**을 추정한다.
3. 아크상 ep05가 전개부 초입이라면 "관계에 균열이 시작될 때"임을 판단한다.
4. 역산 5원칙(`references/recall-design.md`)에 따라 이번 화 사건·감정·훅을 설계하고 원고를 쓴다.
5. `..._EP05_...md` 저장 + 현황표 ep05 행을 ✅로 갱신(불변식).

### 케이스 B — "이 소재로 웹툰 시리즈 시작하고 싶어"

1. 기획서 파일이 없으므로 신규 기획.
2. 독자 페르소나 → 세계관 → 캐릭터(비주얼 시트·일관성 앵커) → 아크·훅 전략을 설계.
3. `[작품제목]_기획서.md` 저장, 맨 아래 빈 에피소드 현황표 삽입.
4. 이후 회차는 케이스 A 흐름을 탄다.

## 7. 일관성 앵커 (표류 방지의 핵심)

캐릭터마다 **화마다 반드시 고정할 1~2가지**를 정해 기획서에 못박는다(예: "은발 + 왼쪽 눈 밑 점", "치즈태비 + 빨간 방울 목걸이"). 작화 단계에서 이게 흔들리면 독자는 다른 캐릭터로 인식한다. 앵커는 비주얼 시트와 함께 기획서 캐릭터 절에 둔다(`references/bible-template.md`). 회차 원고에는 그 화에 등장하는 인물의 앵커만 요약해 옮긴다.

## 8. 주의사항

- **상태 관리 전담.** 이 스킬은 컷 설계·작화를 하지 않는다. 컷 대본은 `story-webtoon-episode`, 작화는 `story-webtoon-art`.
- **비주얼 시트는 본문과 분리.** 캐릭터·공간 외형 디테일은 기획서 캐릭터 절과 에피소드 원고의 비주얼 항목에 모은다 — 스토리 본문에 시각 정보를 나열하지 않는다.
- **게스트는 간략하게(단, 그릴 수 있게).** 여러 화 일관성 부담이 없으므로 전체 기획서엔 이름·역할만, 그 화 외형 상세는 에피소드 원고에.
- **옴니버스도 관통 주제가 있어야 한다.** "그냥 에피소드 모음"은 독자 이탈이 빠르다.
- **파일명 규약 준수.** 공백→언더바, EP## 2자리, 금지문자 제거(`references/filename-convention.md`).
- **재호출 시 요약 복사 금지.** 전체 흐름에서 역산한다(`references/recall-design.md`) — 이 스킬의 가치는 전체 흐름을 읽는 것이다.

## 9. 텍스트 산출 후 검수 체인

회차 원고는 텍스트 산출물이므로, 완성 후 정규 검수 체인을 태운다.

```
원고 초고 → moai-coworker:general-ai-slop-reviewer (1차 일반 후처리)
          → moai-writer:general-humanize-korean (2차 한국어 정밀 윤문)
```

윤문은 **의미·고유명사·수치·인용을 보존**하며 AI 티만 제거한다(내용 재작성 아님).

## 10. 관련 스킬

| 관계 | 스킬 | 역할 |
|------|------|------|
| Before/After | `story-webtoon-planner` | 웹툰 스토리 창작 — 이 원장이 상태를 공급 |
| Before/After | `story-webnovel-writer` | 웹소설 창작 — 동일 원장 메커니즘 |
| 규격 참조 | `story-webtoon-spec` | 데뷔 경로·회차 분량·수익화 훅 확인 |
| 식자 참조 | `story-webtoon-lettering` | 회차 원고의 말풍선·서체·SFX |
| 연속성 감사 | `story-continuity-auditor` (에이전트) | 다회차 캐릭터·플롯·설정 연속성(Soul-ID 포함)을 원장 대비로 검증 |
| Post-검수 (텍스트 산출) | `moai-coworker:general-ai-slop-reviewer` → `moai-writer:general-humanize-korean` | 원고 텍스트 1차 후처리 → 2차 한국어 정밀 윤문 |
