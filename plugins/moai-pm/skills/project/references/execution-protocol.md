# execution-protocol.md — 스킬 체인 실행 프로토콜 (project 스킬)

## 개요

project 스킬이 설계한 스킬 체인을 실행하고, 산출물을 검증하고 전달하는 프로토콜이다.

---

## 전체 실행 플로우: Interview → Plan → Confirm → Execute

```
0. [Evaluate] 요청 평가 사다리로 응답 층을 판단 — 대화형 답변으로 충분하면 여기서 종료
1. [Interview] 소크라테스 인터뷰로 빠진 맥락 수집 (context-collector.md)
2. [Summary] 수집한 맥락을 구조화하여 확인 요약
3. [Plan+Confirm] 스킬 체인 실행 계획 + 완료 기준 제시 → AskUserQuestion으로 승인
4. [Execute] 체인 순차 실행 → 각 단계 요약 보고 → 품질 검증 → 「확인 필요 항목」 명시 → 전달
5. [Post-Execution] 수정 요청 반영(별도 자동 평가 절차 없음)
```

스킵 조건: 단순 조회, 이미 상세 지시, "빠르게" 명시, 후속 요청.

---

## 0. 디렉터 인터랙션 가이드

project 스킬은 사용자에게 진행 중인 작업을 투명하게 보고한다. 멀티스텝 체인은 착수 전 계획을 요약하고, 각 단계 완료 시 짧게 보고한다. 사용자가 "빠르게" 또는 이미 상세히 지시했으면 안내를 생략하고 바로 실행한다.

---

## 1. 실행 전 준비 (Pre-Execution)

### 1-1. 스킬/체인 식별 및 로딩

설계된 체인의 각 스킬이 실행 시점에 로드 가능한지(인벤토리에 존재하는지) 확인한다. 없으면 Gap Detection으로 넘긴다(§Plugin Inventory Scan — SKILL.md 참조).

### 1-2. 복잡도 판단

복합 요청이거나 3개+ 직원 체인이 걸치면 `ultrathink`(심층 분석)로 최적 설계를 결정한다.

### 1-3. 소크라테스식 전제 검증 (규제/전략 산출물)

법률·세무·재무·정부지원 등 DEEP 등급 산출물은 실행 전 전제(가정)를 사용자와 확인한다: "이 판단의 전제가 되는 사실 관계를 확인해도 될까요?"

---

## 2. 워크플로우 실행

### 2-1. 스킬별 체인 실행

각 체인 단계를 순서대로 실행하고, 단계 완료마다 간단히 보고한다(장황한 진행 서술은 피한다).

### 2-2. 에러 처리

체인 중 스킬이 실패하면: (1) 실패 원인을 식별, (2) 대체 경로가 있으면 재시도, (3) 없으면 사용자에게 보고하고 다음 단계 진행 여부를 확인한다.

---

## 3. 산출물 전달

전달 직전 「확인 필요 항목」을 모아 명시한다(출처 미확보 수치·미확인 사실·외부 입력 대기 항목).

---

## 3.5 산출물 품질 검증 루프 (자동)

### 검증 의무 (HARD)

파일 산출물(HWPX/DOCX/PPTX/XLSX)은 전달 전 `quality-evaluator.md`의 Layer 1(파일 유효성) 검증을 통과해야 한다. 텍스트 산출물은 §6 AI 슬롭 후처리(`general-ai-slop-reviewer`)를 통과해야 한다.

상세 검증 레이어(파일 유효성 · 마크다운 렌더링 · AI 작문 패턴 · 근거 검증)는 `quality-evaluator.md` 참조.

---

## 5. 사후 처리 (Post-Execution)

산출물 전달 후, 사용자가 수정을 요청하면 반영한다. 별도 자동 평가 절차는 없다. 피드백은 `/project evolve` 실행 시에만 소크라테스식으로 수집한다(`evaluation-protocol.md` §9).

---

## 6. 실행 경로 선택

### 6-1. 빠른 실행 (Quick Mode)

단순 조회·재구성 요청은 인터뷰를 생략하고 즉시 실행한다.

### 6-2. 대화형 실행 (Interactive Mode, 기본)

일반 산출물은 인터뷰 → 계획 → 확인 → 실행의 표준 플로우를 따른다.

### 6-3. 배치 실행 (Batch Mode)

동일 유형 다건 요청은 1회 확인 후 반복 실행한다.

### 6-4. 검증 깊이 사다리 (위험 비례)

검증 깊이는 요청의 위험 수준에 비례한다. 저위험 작업에 과한 검증을, 고위험 작업에 부족한 검증을 거는 것을 둘 다 막는다.

| 등급 | 대상 | 검증 범위 |
|------|------|----------|
| **QUICK** | 단순 조회·요약·재구성·아이디어 | Layer 1만. 근거 게이트 스킵 |
| **NORMAL** (기본) | 일반 보고서·카피·문서·기획 | Layer 1~2.5. 근거 게이트 활성 |
| **DEEP** | 규제·법률·세무·재무·정부지원·계약·의료, 또는 2개+ 플러그인 체인 | 전체 검증 + 근거 게이트 필수 + 면책 문구 필수 + 전달 전 AskUserQuestion 확인 |

**DEEP 자동 트리거 키워드(seed)**: 법률·계약·개인정보·컴플라이언스·정부지원·세무·재무제표·IR·투자·의료·규제 + 산출물이 2개+ 플러그인 체인에 걸칠 때. 사용자가 "빠르게"·"그냥" 명시 시 QUICK으로 강등하되, DEEP 도메인은 위험 고지 후에만 강등한다. 키워드는 seed일 뿐 — 맥락 판단을 우선한다.

### 6-5. 검색 스케일링 (외부 사실 확인)

<!-- @MX:NOTE: [AUTO] 검색 스케일링은 §6-4 검증 깊이 사다리의 보완 축이다 — 6-4가 "얼마나 깊게 검증할 것인가"를 정하면, 6-5는 "몇 번 찾을 것인가"를 정한다. -->

체인 실행이 외부 사실 확인(WebSearch/WebFetch)을 요구할 때, 도구 호출 횟수는 요청 복잡도에 비례한다.

| 요청 유형 | 호출 규모 | §6-4 정렬 |
|-----------|----------|-----------|
| 단일 사실·시세·현직 확인 | **1회** | QUICK |
| 비교·정책·검증 필요 주제 | **3-5회** | NORMAL |
| 심층 리서치·다출처 종합 | **5-10회+** | DEEP |
| 확립된 개념·역사적 사실·불변 정의 | **검색 생략** — 내부 지식으로 답변 | QUICK 이하 |
| 최신성이 결과를 좌우하는 정보 | **반드시 검색** — 내부 지식 단정 금지 | 유형별 규모 적용 |

검색 결과 수치·날짜·규정은 §3.5 근거 게이트를 그대로 통과해야 한다. 첫 1-2회 검색으로 답이 확정되면 남은 예산을 소진하지 않는다. 검색 과정을 사용자에게 장황하게 서술하지 않는다.

---

## 7. 코드 실행 활용

데이터 검증, 포맷 변환, 결과 필터링 등은 bash/Python 스크립트로 처리한다. 모든 결과를 컨텍스트에 로드하지 않고, 최종 결과만 반환한다.
