# 에디토리얼 덱 독트린 — 13-슬롯 어휘 + 액션타이틀 카피

발표급 덱은 "레이아웃을 발명하지 않고 카피를 발명한다". 이 문서는 (1) **폐쇄형 레이아웃 어휘 + overflow=0 규율**과 (2) **에디토리얼 카피 독트린** 두 축을 office-html-slide의 기존 9 아키타입에 **보완(complement)**으로 얹습니다. 대체가 아닙니다.

---

## 1. 13-슬롯 폐쇄형 레이아웃 어휘 + overflow=0 규율

발표급 덱은 매 슬라이드마다 새 레이아웃을 짜지 않습니다. 프레젠테이션 실무에서 공통적으로 쓰는 **표준 편집형 레이아웃 어휘(아래 13종) 중 하나로만** 짓습니다.

```
cover · chapter · statement · bullets · two-col · cards-3 · data-split ·
image-full · image-split · flow · chart-bar · kpi-3 · closing
```

**형태는 계약, 내용만 채운다.** enum 밖 레이아웃을 발명하는 순간 덱의 시각 일관성이 무너집니다. 각 슬롯은 고정 필드 + `max_chars`/`max_lines`를 가진다고 보고, 그 안에서 카피만 채웁니다.

**overflow=0 규율** — 실폰트(Pretendard) 기준 카피가 박스폭 **90%를 넘으면 카피를 줄입니다**(폰트를 줄이거나 박스를 키우지 않음). 오버플로 0이 목표이며, 이는 [`deck-quality-rubric.md`](deck-quality-rubric.md) 기준 #28(텍스트 오버플로 hard-fail)과 직결됩니다. "글이 안 들어가면 글을 줄인다"가 원칙이지 "폰트를 줄인다"가 아닙니다.

### 기존 9 아키타입과의 매핑 (보완 관계)

office-html-slide는 이미 9 아키타입(Title/Agenda/Problem/Solution/Features/Stats/Team/CTA/Closing, [`deck-manuscript-schema.md`](deck-manuscript-schema.md) 참조)을 씁니다. 13-슬롯은 이를 **대체하지 않고**, 각 아키타입을 더 세분화된 시각 슬롯으로 내려주는 보완 어휘입니다.

| 13-슬롯 | 대응 9 아키타입 | 보완 포인트 |
|---|---|---|
| cover | Title | 풀블리드 히어로 + 텍스트측 스크림 |
| chapter | Agenda(섹션 구분) | 큰 섹션 번호 + 한 줄 |
| statement | Problem/Solution | 지배 문장 1개(단일 메시지) |
| bullets | Agenda/Features | 불릿 ≤6줄 규율 |
| two-col | Features(비교) | 카드 아님 — 세로 1px hairline 분할 |
| cards-3 | Features | 3열 타일 |
| data-split | Stats | 좌 텍스트 / 우 히어로 격리 |
| image-full / image-split | Problem/Solution | 이미지 깊이(스크림/글로우) |
| flow | Solution/Features | 파이프라인 단계 |
| chart-bar | Stats | 인라인 SVG 차트 |
| kpi-3 | Stats | 거대숫자 3단 위계 |
| closing | Closing/CTA | 마무리 단서 1줄 |

> 매핑이 애매한 복합 슬라이드는 가장 가까운 슬롯을 골라 **하나로 고정**합니다. 두 슬롯을 한 슬라이드에 섞는 것이 overflow와 시각 혼란의 원인입니다.

## 2. 에디토리얼 카피 독트린

레이아웃이 계약이라면, 카피는 **완결된 서술**이어야 합니다. 발표급 카피의 4대 규칙:

### 2-1. 액션타이틀 = 마침표로 끝나는 완결 서술문

제목 자체가 핵심 결론을 담은 **완결 문장**이어야 하며, 독자가 본문을 해석해 결론을 추론하게 해서는 안 됩니다(Minto 피라미드가 슬라이드로 내려온 형태).

- **명사 나열 제목 금지**: "품질 게이트" (X) → "미달은, 통과하지 못한다." (O)
- 최대 15단어·2줄 이하, 능동태, 마침표로 끝냄
- 예: "두 달 만에 1억 명." / "복붙을, 위임으로 바꿨다."

### 2-2. Assertion-Evidence 모델

제목은 **주장(assertion)**, 본문은 불릿 나열이 아니라 **시각적 증거(evidence)**(차트/이미지/다이어그램/거대숫자). 슬라이드 1장 = 지배 오브젝트 1개.

### 2-3. One-Message-Per-Slide

슬라이드 하나에 통찰 정확히 하나. **제목에 등위접속사("및/와/과/그리고")가 있으면 메시지가 둘이라는 위반 신호**입니다. 본문에 서로 다른 결론성 완결문이 2개 이상이면 슬라이드를 쪼갭니다.

### 2-4. 에디토리얼 골격 (카피 조립 순서)

```
eyebrow (대문자·accent, 앞에 짧은 rule)
  → 액션타이틀 (마침표로 끝나는 단정문)
    → 지배 오브젝트 1개 (stat / chart / flow / image …)
      → tail (hairline 위 단서 1줄, 선택)
```

이 골격 밖의 카피 배치(제목 아래 소제목 아래 또 소제목…)는 위계를 무너뜨립니다.

## 3. 의무 슬롭 체인과의 관계 (보완)

office-html-slide는 모든 카피에 **의무 슬롭 체인**(`general-ai-slop-reviewer` → `general-humanize-korean`)을 적용합니다([`SKILL.md`](../SKILL.md) §7). 그 체인은 **부정형 탐지**(나쁜 패턴 제거)에 특화되어 있고, 특히 구조적 슬롭 S1 3종을 잡습니다:

1. **대시 대비 헤드라인** — "X — Y" 대시 분할
2. **조사·체언 종결 조각문** — 서술어 없이 조사/체언으로 끝남
3. **"A에서 B로" 전환 공식** — "엑셀에서 노션으로" 류 도입

**본 독트린은 그 체인이 갖지 못한 "긍정형 규칙"을 더합니다** — 슬롭 체인이 *하지 말 것*(대시·조각문·전환공식)을 지운다면, 본 문서 §2-1의 액션타이틀 규칙은 *할 것*(마침표로 끝나는 완결 서술문)을 지정합니다. 두 축은 상보적입니다:

| | 슬롭 체인(S1 3종) | 본 독트린(§2-1) |
|---|---|---|
| 방향 | 부정형(제거) | 긍정형(생성) |
| 대상 | 조각문·대시·전환공식 | 완결 서술 액션타이틀 |
| 예 | "성공의 열쇠 — 자동화" 제거 | "자동화가 열쇠였다." 생성 |

슬롭 체인이 조각문 "성공의 열쇠 — 자동화"를 잡아 지운 자리에, 본 독트린이 완결문 "자동화가 열쇠였다."를 채웁니다. **슬롭 체인은 여전히 의무이며, 본 독트린은 그 위의 긍정형 보완 규칙입니다.**

카피 규칙은 공개된 프레젠테이션 실무 원칙에서 가져왔습니다: Barbara Minto 피라미드 원칙(결론 먼저 → 액션타이틀), one-message-per-slide 원칙, Michael Alley(Penn State)의 assertion-evidence 모델. 13종 레이아웃 어휘는 프레젠테이션 도구(reveal.js·Marp·PowerPoint 레이아웃 등)에서 공통으로 쓰는 표준 슬라이드 유형을 정리한 것입니다.
