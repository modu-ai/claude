# `.moai/` — MoAI Working Directory

This directory is the project-local working area managed by the `moai` tool and
the SPEC-driven development workflow. It is created when a project is initialized
(`moai init`) and is intended to live alongside the project's own source under
version control — except for the volatile paths noted below, which the generated
`.gitignore` excludes.

The structure is **language-neutral**: it does not assume any particular
programming language. Every supported language is treated equally — the same
layout applies whether the project is written in Go, Python, TypeScript, Rust,
Java, Kotlin, C#, Ruby, PHP, Elixir, C++, Scala, R, Dart/Flutter, Swift, or
JavaScript.

## Standard subdirectories

### Shipped by the template

| Directory    | Purpose                                                                  |
|--------------|--------------------------------------------------------------------------|
| `config/`    | MoAI configuration (YAML sections: language, quality, harness, etc.)    |
| `project/`   | Project-level documentation (`product.md`, `structure.md`, `tech.md`, brand) |
| `design/`    | Design-system working artifacts (briefs, tokens, wireframes, screenshots)|
| `docs/`      | Long-form reference docs authored for this project (guides, setup notes) |
| `decisions/` | Architectural decision records (ADRs)                                    |
| `branches/`  | Branch-origin decision audit trail                                       |
| `evolution/` | Changelog, learning telemetry, new-skill proposals                       |
| `learning/`  | Persistent study notes generated by the learning output style            |
| `reports/`   | Generated analysis reports (plan-audit, sync, trust-5, etc.)             |
| `logs/`      | Runtime logs (gitignored — regenerated)                                  |
| `state/`     | Runtime state snapshots, e.g. context-usage (gitignored — regenerated)   |

Plus `status_line.sh` (rendered from `status_line.sh.tmpl`) — the status-line script.

### Generated at runtime (not in the template)

| Directory | Purpose                                                                   |
|-----------|---------------------------------------------------------------------------|
| `specs/`  | SPEC documents — one directory per SPEC, named `SPEC-<DOMAIN>-<NNN>/`.    |
|           | See **`.moai/specs/` root integrity** below for the placement contract.   |
| `plans/`  | Planning, roadmap, and strategy documents. **The correct home for         |
|           | ROADMAP-style docs** — never put them in `specs/`.                        |
| `brain/`  | Idea-capture workspace                                                    |
| `bodp/`   | Branch-origin decision persistence                                        |
| `harness/`| Harness learning subsystem data                                           |
| `cache/`  | Transient caches (gitignored)                                             |
| `temp/`   | Scratch files (gitignored)                                                |

## Where does X go?

| Artifact                         | Location                                  |
|----------------------------------|-------------------------------------------|
| A SPEC and its artifacts         | `specs/SPEC-<DOMAIN>-<NNN>/`              |
| Roadmap, planning, strategy docs | `plans/` or the project root `docs/`      |
| Project product/structure/tech   | `project/`                                |
| Design brief / tokens            | `design/`                                 |
| Decision record (ADR)            | `decisions/`                              |
| Study notes                      | `learning/`                               |
| Research notes for an in-flight SPEC | inside that SPEC's directory           |

## `.moai/specs/` root integrity

The `specs/` directory root holds **only** per-SPEC working directories named
`SPEC-<DOMAIN>-<NNN>/`. Do **not** place loose files (e.g. `ROADMAP.md`,
`README.md`, `NOTES.md`) or non-`SPEC-` directories at the `specs/` root — those
belong in `plans/` or `project/`. Entries prefixed with `_` (e.g. `_archive/`)
are the exempt tooling convention for retired SPECs.

`moai spec lint` reports any non-SPEC root entry as a `SpecsDirForeignEntry`
warning. See the SPEC frontmatter schema rule (`development/spec-frontmatter-schema.md`
§ `.moai/specs/` Root Integrity) for the full integrity contract.

## Volatile paths (gitignored)

`logs/`, `state/`, `cache/`, `temp/`, and `reports/*.md` are regenerated at
runtime and excluded from version control by the generated `.gitignore`. Keep
regenerable artifacts out of tracked paths; check in only source-of-truth
documents (SPECs, project docs, decisions, config).
