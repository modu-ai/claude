# story-webnovel-planner: 웹소설 연재 기획

웹소설의 **연재 판**을 고르는 스킬이다. 플랫폼 선택이 회차 분량·절단 강도·연참 주기·수익 구조를 전부 결정하므로, 집필(`story-webnovel-writer`)보다 먼저 온다. 플랫폼을 잘못 고르면 잘 쓴 회차도 독자를 만나지 못한다.

## 1. 개요

이 스킬은 **"어디에, 어떤 구조로 연재할 것인가"**라는 선행 판단에 집중한다. 6개 플랫폼은 회차 분량 관행·결제 방식·독자층·독점 조건이 서로 크게 다르다. 이 선택을 확정한 뒤에야 한 회를 어떻게 쓰고 어디서 자를지(`story-webnovel-writer`)를 정할 수 있다.

## 2. 이 스킬이 하지 않는 것

- **회차 집필 안 함** — 실제 원고·장르 문법·절단 강도 설계는 `story-webnovel-writer` 소관.
- **정산율·글자 수를 지어내지 않음** — 구체 수치는 플랫폼 정책이라 시기별로 바뀐다. 확인 안 된 수치는 `[미확인 — 플랫폼 공식 안내 확인 필요]`로 표기하고 확인처를 병기한다.
- **파일 저장·상태 갱신 안 함** — 마스터 기획서 저장과 회차 현황표는 `story-series-bible`가 소유한다.
- **계약 심화 자문 안 함** — 표준계약서 조항·법률 해석은 `moai-lawyer` 라우팅.

## 3. 사전 확인 — 규격 먼저 고정

기획에 들어가기 전에 플랫폼 규격을 먼저 확인한다.

- `references/webnovel-platforms.md` — 6개 플랫폼 비교표(회차 분량·결제 구조·태그·독자층·프로모션 진입·독점 여부).
- `references/serialization-contract.md` — 선독점/독점 계약, 정산 구조, 유료화 전환 기준(수치는 출처 병기, 심화는 `moai-lawyer`).
- `${CLAUDE_PLUGIN_ROOT}/skills/story-webtoon-spec/references/monetization-hooks.md` — 기다무·미리보기 유료 전환·사이다/고구마 축(웹툰화 연계 시).

> ⚠️ 정산 비율·회차 최소 글자 수 같은 구체 수치는 단정하지 않는다. 플랫폼 공식 안내(작가 센터/투고 가이드)에서 확인 후 반영하고, 미확인이면 플래그를 유지한다.

## 4. 워크플로우

### Step 1 — 작품 성향 진단

장르·독자 성별 지향·핵심 소구(사이다/힐링/몰입)를 정리한다. 이 세 축이 플랫폼 적합도를 가른다(예: 남성향 현대판타지 신규 연재의 핵심 무대와 여성향 로맨스판타지 무대는 다르다).

### Step 2 — 플랫폼 선택 (6개 비교)

`webnovel-platforms.md` 표를 근거로 타깃 플랫폼을 1개(또는 선독점→확장 순서)로 고른다. 비교 축:

- **회차 분량 관행** — 편당 결제 단위인지, 스크롤 단위인지.
- **결제 구조** — 편당 결제 / 기다리면 무료(기다무) / 무료 연재 후 유료 전환.
- **태그 체계** — 검색·랭킹 노출이 태그로 결정되는가.
- **독자층** — 성별·연령·선호 장르.
- **프로모션 진입 조건** — 유료화·프로모션 배너에 오르는 진입 문턱.
- **독점 여부** — 독점/선독점/비독점 선택지.

> 구조 지식(편당 결제 vs 기다무, 독점 vs 비독점, 성인/일반 구분)은 사실로 서술한다. 정산율·최소 글자 수 같은 수치만 `[미확인]` 원칙을 적용한다.

### Step 3 — 연재 구조 3분류

플랫폼을 고른 뒤 연재 구조를 정한다. 구조가 회차 밀도·절단 리듬을 좌우한다.

| 구조 | 특징 | 적합 |
|------|------|------|
| **에피소드 독립형** | 각 회 완결에 가까움 | 옴니버스·생활물 |
| **연속 서사형** | 하나의 큰 서사가 이어짐 | 현판·로판·무협 주류 |
| **혼합형** | 회 단위 소절정 + 관통 서사 | 대부분의 유료 연재작 |

> ⚠️ 옴니버스도 관통 주제가 없으면 이탈이 빠르다 — 한 문장 주제를 반드시 정한다.

### Step 4 — 레귤러/게스트 인원 예산

매 회 등장하는 핵심 인물(레귤러)과 특정 구간만 등장하는 게스트를 계층으로 나눈다. 레귤러가 과다하면 독자가 관계를 따라가지 못한다 — 초반 레귤러는 소수로 시작하고 확장은 서사 필요에 따라 연다.

### Step 5 — 태그·프로모션 전략

선택 플랫폼의 태그 체계에 맞춰 노출 태그를 배치하고, 프로모션 진입 조건(연재 회차 수·조회·유료 전환 시점)을 목표로 역산한다. 수치 목표는 플랫폼 정책 확인 후 채운다.

## 5. 출력 형식

기획 내용은 마스터 기획서 형식으로 `story-series-bible`에 넘겨 저장한다.

```markdown
# [작품제목] 웹소설 연재 기획 (초안)

## 작품 성향
- 장르 / 지향 독자 / 핵심 소구

## 타깃 플랫폼
- 선택: [플랫폼] (근거: webnovel-platforms 표)
- 결제 구조 / 독점 여부 / 프로모션 진입 조건
- [미확인 항목은 플래그 + 확인처 병기]

## 연재 구조
- 분류: [독립형 / 연속형 / 혼합형]
- 관통 주제 1문장: ___

## 인원 예산
- 레귤러 N명 / 게스트 계층

## 태그·프로모션
- 노출 태그 / 프로모션 목표 시점
```

→ 저장·회차 현황표 생성은 `story-series-bible`로 넘긴다.

## 6. 주의사항

- **수치 환각 금지.** 정산율·글자 수·프로모션 문턱은 확인 전까지 `[미확인]`.
- **플랫폼 먼저, 집필 나중.** 플랫폼을 안 고르고 회차부터 쓰면 분량·절단이 표류한다.
- **독점 결정은 되돌리기 어렵다.** 선독점/독점 조건은 계약이므로 `serialization-contract.md` + `moai-lawyer` 확인.
- **트리거 6개 플랫폼 모두 커버.** 특정 플랫폼만 걸면 다른 플랫폼 사용자가 호출 못 한다.
