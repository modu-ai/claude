요약 (Executive Summary)

현재 marketplace moai-claude v6.2.0은 18개 플러그인으로 구성된다. 원래 moai-coworker가 단일 플러그인에 담고 있던 직군별 스킬을 전문가 플러그인 8종(moai-writer, moai-marketer, moai-seller, moai-officer, moai-lawyer, moai-accountant, moai-recruiter, moai-tutor)으로 분리했고, v6.1.0에서 CS·컨설턴트·커리어 계열 3종(moai-cs, moai-consultant, moai-career)을 추가했다. v6.2.0에서는 이질적 도메인 재분리를 통해 moai-story(스토리/IP 13종, writer에서 분리)·moai-analyst(데이터/공공데이터 11종, officer에서 분리)·moai-media(미디어 생성 9종, marketer에서 분리) 3종을 신설했고, 라이프스타일·자기계발 7종은 moai-coworker general로 회수했다. 그 결과 moai-writer(10)·moai-officer(13)·moai-marketer(19)는 슬림화되고 moai-coworker는 32스킬로 확장됐다 (스킬 총수 233 불변 — 재배치만). 직군별 사용자는 이제 자신에게 필요한 스킬만 해당 전문가 플러그인에서 설치하면 된다.

설계 근거는 2026-07-09에 수행된 두 건의 실측 테스트다. 보고서 1(Cowork 원격 샌드박스)은 에이전트 16종 중 15종의 정상 동작과 3계층 권한 구조(full-power / worker / read-only audit)의 정의 레벨 강제를 확인했고, 보고서 2(Claude Desktop Code 런타임, Claude Code 2.1.205 / macOS)는 Skills·Subagents·번들 MCP·Hooks·settings.json·dynamic Workflow의 동작과 함께 LSP·output-styles·Commands의 패리티 갭을 식별했다. 전문가 플러그인은 검증된 메커니즘(skills + agents + 선택적 .mcp.json)만 사용하도록 설계해 확인된 모든 갭을 회피한다.

전체 설계는 Agent Harness Engineering 체계 — 에이전트가 루프를 돌며 스킬을 사용하고 오케스트레이션으로 목적을 달성하는 AI 실행 구조 — 를 따르며, 프레임워크의 5대 핵심 가치를 계획 요소에 직접 매핑한다(§04).

① Reliability 신뢰성 ② Safety 안전성 ③ Scalability 확장성 ④ Observability 관측성 ⑤ Maintainability 유지보수성

실행은 Phase 0(거버넌스)부터 Phase 4(최종 등록·검증)까지 5단계로 진행하며, 각 Phase는 스킬 로드 스모크 테스트 · 에이전트 스폰 카나리아 · cross-reference grep 0건 · marketplace.json 검증의 4중 게이트를 통과해야 다음 단계로 넘어간다. 시간 추정은 사용하지 않고 우선순위 라벨(Priority High/Medium)과 단계 순서만으로 관리한다.

04

설계 원칙 — 하네스 엔지니어링 설계 체계 (Agent Harness Engineering)

본 계획의 설계는 Agent Harness Engineering 체계를 따른다. 핵심 명제: 에이전트(AGENT)가 루프를 돌며 스킬(SKILL)을 사용하고, 오케스트레이션(ORCHESTRATION)으로 목적을 달성하는 AI 실행 구조. 아래에서 프레임워크의 3요소·에이전트 루프·오케스트레이션 엔진· 가드레일·5대 핵심 가치를 본 계획의 구성 요소에 1:1로 매핑한다. 모든 매핑은 §02 테스트 근거에 기반한다.

4.7  실행 원칙 4종 (테스트 근거 기반)

PRINCIPLE 1

비개발자 전문가 플러그인 = skills + agents 전용

hooks · LSP · output-styles 의존을 두지 않는다. 이는 보고서 2에서 확인된 데스크탑 패리티 갭 전부(LSP 미동작, output-styles 활성화 미확인, Commands의 스킬 표면화)를 원천 회피하는 구성이다. 도메인 MCP가 존재하는 경우에 한해 .mcp.json을 선택 번들한다 — 번들 MCP는 korean-law 실데이터 반환으로 동작이 입증됐다.

근거: 보고서 2 (Skills ✅ · Subagents ✅ · 번들 MCP ✅ / LSP ❌ · output-styles ⚠️ · Commands 🟡)
PRINCIPLE 2

2-에이전트 표준 세트 — 검증된 3계층 권한 모델 준수

모든 전문가 플러그인은 (1) worker-tier 에이전트 (Read/Write/Edit/Bash/Glob/Grep/WebSearch/WebFetch/Skill)와 (2) read-only auditor 에이전트 (Read/Grep/Glob/Bash만 — sync-auditor 패턴)를 1쌍 탑재한다. 정의 레벨에서 Write/Edit를 제거하면 read-only가 실제로 강제된다는 점이 보고서 1에서 확인됐다.

근거: 보고서 1 (3계층 권한 구조 — 정의 레벨 강제 확인)
PRINCIPLE 3

스킬 본문 이동(move) — 복사 금지, 네임스페이스 접두사 불변

스킬 본문은 moai-coworker에서 새 플러그인으로 이동하며 복사본을 남기지 않는다. coworker는 범용 business/office 코어를 유지한다. 각 새 플러그인 내부에서 스킬 접두사는 변경하지 않는다 — 대량 rename은 하지 않는다.

근거: 한국어 슬롭 감사 교훈 — 대량 rename은 참조 파손(broken reference)을 양산
PRINCIPLE 4

검증 방법론 = 보고서 2 카나리아 방식 재사용

플러그인 scaffold 완료 시마다 해당 worker 에이전트를 스폰해 카나리아 파일 + 타임스탬프를 쓰게 하고, 파일 트레이스를 독립 검증한다. 모든 스킬은 skill listing으로 로드 테스트한다.

근거: 보고서 2 (Subagents file-trace canary 검증 방식)

코더(moai) .lsp.json 선언 — 공식 포맷 기준 5종

공식 .lsp.json 포맷(서버명 → 설정 매핑, 필수 command + extensionToLanguage)에 따라 보고서 2 실측 설치 세트 5종을 선언한다: 

{
  "go":         { "command": "gopls",                      "args": ["serve"],   "extensionToLanguage": { ".go": "go" } },
  "python":     { "command": "pyright-langserver",         "args": ["--stdio"], "extensionToLanguage": { ".py": "python" } },
  "rust":       { "command": "rust-analyzer",                                   "extensionToLanguage": { ".rs": "rust" } },
  "swift":      { "command": "sourcekit-lsp",                                   "extensionToLanguage": { ".swift": "swift" } },
  "typescript": { "command": "typescript-language-server", "args": ["--stdio"], "extensionToLanguage": { ".ts": "typescript", ".tsx": "typescriptreact", ".js": "javascript", ".jsx": "javascriptreact" } }
}
