moai-adk 범용화 전략
— Claude Code 전용에서 멀티 에이전트 CLI 툴킷으로

현재 Claude Code 전용으로 설치·사용 중인 moai-adk 바이너리를, OpenAI Codex·Gemini CLI 등 어떤 코딩 에이전트에서도 동일하게 쓸 수 있도록 구조를 개선하는 방안을 조사했다. 결론부터: 코드의 절반 이상은 이미 LLM 중립적이며, 업계에는 이 문제를 정확히 풀기 위해 만들어진 3개의 공개 표준(AGENTS.md · Agent Skills · MCP)과 검증된 선례(GitHub spec-kit)가 존재한다.

2026-07-14 · 대상: moai-adk-go (github.com/modu-ai/moai-adk) · 웹 리서치 + 코드베이스 스캔 기반

요약 (TL;DR)

  • 진단: moai-adk-go의 약 55~65%(SPEC 엔진, 품질 게이트, git, LSP/ast-grep, MX 태그)는 이미 하네스 중립 코어다. 나머지 35~45%(hook JSON 프로토콜, settings.json, statusline, 런처, .claude/ 템플릿)가 Claude Code 어댑터다. GLM 백엔드는 모델 교체이지 하네스 추상화가 아니다.
  • 기회: 2025년 12월 Anthropic이 Agent Skills를 공개 표준화한 뒤 Codex·Gemini CLI·Copilot·Cursor 등 30개 이상 도구가 채택했다. 스킬은 .agents/skills/에 한 번 작성하면 대부분의 에이전트에서 그대로 동작한다. 지침 파일은 AGENTS.md(Linux Foundation 관할, 6만+ 저장소 채택)가 사실상 표준이다.
  • 권장 구조: ① CLI-first 코어(어떤 에이전트든 셸로 moai gate를 호출 — 최소공배수 어댑터) ② 표준 자산 레이어(스킬은 Agent Skills 표준으로, 지침은 AGENTS.md 정본 + CLAUDE.md는 @AGENTS.md 임포트) ③ spec-kit 스타일 하네스 어댑터 레지스트리(moai init --agent claude|codex|gemini) ④ 선택적 moai mcp serve.
  • 한계 인지: hooks·output styles·서브에이전트·AskUserQuestion은 Claude Code 고유 기능이라 이식이 안 된다. 전략은 "하향 평준화"가 아니라 점진적 향상(progressive enhancement) — 코어는 어디서나, 풀 오케스트레이션은 Claude Code에서.

1현재 moai-adk 구조 진단

쉽게 말하면 — moai-adk는 "엔진"과 "차체"로 나뉜다. 엔진(SPEC 워크플로우, 품질 검사, git 자동화)은 어떤 차에나 얹을 수 있는데, 지금은 차체(Claude Code 연결부)에 용접돼 있는 상태다. 용접을 볼트 체결로 바꾸는 것이 이번 과제다.

moai-adk-go 저장소를 읽기 전용으로 스캔한 결과:

LLM 중립 코어 (~55–65%)

  • internal/spec — SPEC 워크플로우
  • internal/hook/quality·security, foundation/trust — 품질·보안 게이트
  • internal/git, worktree — git 자동화
  • internal/lsp, astgrep, mx — 진단·주석
  • internal/ciwatch, update, migration

Claude Code 어댑터 (~35–45%)

  • internal/cli/hook.go + internal/hook 디스패처 — Claude hook JSON stdin/stdout 프로토콜에 결합
  • internal/settings — .claude/settings.json 병합 엔진
  • internal/statusline — Claude statusLine 계약
  • cc.go / cg.go / glm.go / launcher.go — Claude Code 실행기
  • internal/template/templates/.claude/* — 에이전트·스킬·규칙 템플릿
중요한 발견 — 이미 존재하는 moai glm / moai cg의 멀티 백엔드 지원은 Claude Code 하네스 안에서의 모델 공급자 교체다(Anthropic 호환 엔드포인트로 GLM을 소비). Codex CLI나 Gemini CLI 같은 다른 하네스를 위한 어댑터 인터페이스는 아직 없다. 다만 unifiedLaunchDefault(mode) 같은 런처 추상화가 이미 있으므로, 하네스 축을 추가할 자리는 마련돼 있다.
구체 예시
품질 게이트 moai gate(lint+format+type+test 병렬 실행)는 셸 명령이므로 이론상 Codex도 지금 당장 호출할 수 있다. 그러나 결과가 Claude hook 이벤트(PostToolUse 등)로 되돌아가도록 설계돼 있어, Codex에서는 "실행은 되지만 하네스에 피드백이 연결되지 않는" 상태가 된다. 코어는 살아 있고, 배선만 Claude 전용인 셈이다.

2생태계 지형: 3대 범용화 표준

쉽게 말하면 — "어느 에이전트에서나 똑같이 쓰기"라는 문제를 업계가 이미 세 조각으로 나눠 표준화했다. 지침(에이전트가 지켜야 할 규칙 문서)은 AGENTS.md, 스킬(재사용 가능한 작업 절차 패키지)은 Agent Skills, 도구(에이전트가 호출하는 외부 기능)는 MCP다. moai-adk가 새 표준을 발명할 필요가 없다.

2-1. AGENTS.md — 지침 파일 표준

  • "에이전트를 위한 README". Linux Foundation 산하 Agentic AI Foundation이 중립 관할하며, 60,000개 이상 저장소, 20개 이상 도구(Codex, Copilot, Cursor, Jules, Amp, Zed, Windsurf 등)가 네이티브로 읽는다.
  • Codex CLI는 AGENTS.md가 1차 지침 파일이며 git 루트→현재 디렉터리로 내려가며 계층적으로 읽는다. 기본 32KiB 제한(project_doc_max_bytes) 초과분은 조용히 잘린다.
  • Claude Code는 CLAUDE.md를 읽는다. 업계 합의 패턴: AGENTS.md를 정본(canonical)으로 두고, CLAUDE.md는 첫 줄 @AGENTS.md 임포트 한 줄짜리로 만든다. 저장소와 함께 이동하므로 팀원 전원이 clone만으로 동일 동작을 얻는다. (역방향 — Codex가 CLAUDE.md를 읽게 하는 project_doc_fallback_filenames — 는 사용자 홈 설정이라 저장소와 함께 이동하지 않아 약하다.)

2-2. Agent Skills — 스킬 표준 (SKILL.md)

  • 2025년 12월 18일 Anthropic이 공개 표준으로 릴리스(agentskills.io, Apache 2.0/CC-BY-4.0). 공개 48시간 안에 Microsoft(VS Code/Copilot)와 OpenAI(Codex)가 지원을 출시했고, 2026년 3월 기준 30개 이상 도구가 채택 — Claude Code, Codex, Gemini CLI, Copilot, Cursor, Goose, JetBrains Junie, Kiro, OpenCode, OpenHands, Roo Code, Mistral Vibe 등 (agentskills.io 클라이언트 쇼케이스에서 직접 확인).
  • 구조: 폴더 + SKILL.md(YAML frontmatter의 name/description + 지시문) + 선택적 scripts/ references/ assets/. 3단계 점진 공개(목록→본문→참조)로 컨텍스트를 아낀다 — moai-adk의 Progressive Disclosure 설계와 정확히 같은 철학이다.
  • 디렉터리 규약: .agents/skills/가 범용 위치. Codex는 ./.agents/skills → 저장소 루트 → ~/.agents/skills → /etc/codex/skills 순으로 탐색하고, 스킬 목록에 컨텍스트의 최대 2%(또는 8,000자)만 쓴다. 벤더 확장(agents/openai.yaml)은 다른 에이전트가 무시하므로 공존 가능.
  • 이식성 실증: Claude Code용으로 쓴 스킬을 Codex 스킬 디렉터리에 복사하면 그대로 동작한다는 것이 커뮤니티에서 반복 확인됐다. 단, Claude 전용 기능(컨텍스트 포크, 서브에이전트 위임)에 의존하는 스킬은 예외.

2-3. MCP — 도구 노출 표준

  • 하나의 MCP 서버를 만들면 Claude Code(.mcp.json), Codex(config.toml + codex mcp 서브커맨드), Gemini CLI(extensions의 mcpServers) 모두에서 소비 가능 — "도구를 한 번 만들어 모든 에이전트에 공급"하는 유일한 공식 통로다.
  • 베스트프랙티스: 도구는 좁고 예측 가능하게(최소 권한), 기본은 읽기 전용, 프로젝트 스코프 설정을 저장소에 커밋해 팀 전체가 clone만으로 동일 도구를 획득, 도구 수가 많아지면 온디맨드 로딩·코드 실행 패턴(Anthropic 권고)으로 토큰을 절약.
  • Gemini CLI는 한 걸음 더 나아가 extensions(manifest gemini-extension.json + GEMINI.md 컨텍스트 + commands/*.toml + MCP 서버 묶음)라는 배포 단위를 제공한다 — moai-adk의 "플러그인" 개념과 거의 일대일 대응.

3선례: GitHub spec-kit의 멀티 에이전트 아키텍처

왜 이 선례인가 — spec-kit은 moai-adk와 같은 "SPEC 기반 개발 방법론 툴킷"이면서, 처음부터 30개 이상 에이전트(Claude Code, Copilot, Gemini CLI, Cursor, Codex CLI…)를 지원하도록 설계됐다. moai-adk가 가려는 길을 이미 가 본 프로젝트다.
  • 통합 레지스트리: 에이전트마다 자기완결적 integration 서브패키지(src/specify_cli/integrations/)를 두고, 베이스 클래스를 형식별로 계층화 — MarkdownIntegration(Cursor류), TomlIntegration(Gemini), SkillsIntegration(Claude Code·Codex), 커스텀(Copilot). 레지스트리가 단일 진실 공급원이고, 새 에이전트 추가는 코드 복제가 아니라 설정 추가로 끝난다.
  • 초기화 어댑테이션: specify init이 선택한 에이전트에 맞는 명령 파일·컨텍스트 규칙·디렉터리 구조를 깔아 준다. 인자 플레이스홀더($ARGUMENTS 등)도 에이전트별 설정에서 가져온다.
  • 커맨드의 이중 형태: 같은 워크플로우가 대부분의 에이전트에선 슬래시 커맨드로, 스킬 지원 플랫폼에선 설치형 스킬로 배포된다(Claude Code는 .claude/skills/, Codex는 $speckit-<command>).
  • 멀티 설치 안전성: 0.8.5부터 여러 에이전트 통합을 한 저장소에 공존 설치 가능 — 격리된 디렉터리 + 충돌 없는 컨텍스트 파일 + 독립 설치 매니페스트가 조건.
  • 설계 철학: "방법론이지 제품 락인이 아니다(methodology, not a product lock-in)" — 에이전트 키를 실행 파일 이름과 일치시켜 shutil.which()로 감지하는 단순한 규약까지, 전부 moai-adk에 이식 가능한 패턴이다.

Layer 0 — CLI-first 코어 (이미 절반 이상 완성)

  • 코어 기능을 하네스를 모르는 순수 CLI 명령으로 정리: moai gate, moai spec lint, moai mx scan 등. 종료 코드 + 구조화된 JSON 출력이 계약.
  • 현재 Claude hook JSON에 결합된 internal/hook 디스패처에서 이벤트 모델을 추상화: "Claude PostToolUse JSON 파싱"과 "파일 변경 후 품질 검사 실행"을 분리하면, 후자는 어느 하네스에서도 재사용된다.

Layer 1 — 표준 자산: 한 번 작성, 어디서나 설치

  • 스킬: 현재 .claude/skills/moai*의 28개 스킬을 Agent Skills 표준 준수 형태로 정리해 .agents/skills/를 정본 위치로 삼는다. Claude Code·Codex·Gemini CLI가 모두 이 위치(또는 복사/링크)를 읽는다. Claude 전용 기능(서브에이전트 위임, AskUserQuestion 언급 등)을 쓰는 스킬은 frontmatter 코어만으로 동작하도록 분리하거나 어댑터 오버레이로 남긴다.
  • 지침: moai init이 AGENTS.md를 정본으로 생성하고, CLAUDE.md는 @AGENTS.md 한 줄 + Claude 전용 계층(rules, output styles 포인터)만 갖게 한다. 32KiB 이하 유지(Codex 절단 한계).

Layer 2 — 하네스 어댑터 레지스트리 (spec-kit 패턴)

  • moai init --agent claude|codex|gemini (기본값 claude, 복수 지정 허용). 어댑터별 책임: 설정 파일 쓰기(settings.json vs config.toml vs gemini-extension.json), 이벤트 배선(Claude hooks는 풀 지원, 타 하네스는 지원 범위 명시), 런처(moai cc처럼 moai cx/moai gm 또는 통합 moai launch --agent).
  • Go에선 spec-kit의 클래스 계층 대신 인터페이스 + 레지스트리 맵이 자연스럽다: type HarnessAdapter interface { Detect() bool; Install(assets) error; WireEvents() error; Launch(opts) error }.

Layer 3 — moai mcp serve (선택적, 후순위)

  • 셸 접근이 약한 환경(IDE 에이전트, 웹 에이전트)이나 승인 정책이 엄격한 조직을 위해 코어 기능을 MCP 도구로 노출. 좁은 도구·읽기 전용 기본·프로젝트 스코프 설정 커밋이라는 MCP 베스트프랙티스를 따른다.
  • 단, CLI가 이미 통하는 터미널 에이전트에게 MCP는 중복이다 — "MCP를 먼저 만들지 말고, CLI로 안 되는 지점이 확인되면 만들라"가 Codex 공식 문서의 레이어링 권고(단순한 층부터 소진)와도 일치한다.
동작 시나리오 (Codex 사용자)
① moai init --agent codex → AGENTS.md 생성, .agents/skills/에 moai 스킬 설치, .codex/config.toml에 프로젝트 스코프 설정 기록.
② 사용자가 Codex에서 $moai-plan "로그인 기능" 호출 → SKILL.md 지시에 따라 Codex가 moai spec new를 셸로 실행, SPEC 문서 생성.
③ 구현 후 스킬 지시에 따라 moai gate 실행 → 종료 코드/JSON으로 품질 판정을 받아 Codex가 다음 행동 결정.
Claude Code에서라면 같은 흐름이 hooks·서브에이전트·AskUserQuestion까지 얹힌 풀 오케스트레이션으로 동작한다 — 이것이 점진적 향상이다.

6마이그레이션 로드맵 (단계 순서)

M1 — 코어/어댑터 경계 확정 (내부 리팩터링)
internal/hook의 이벤트 파싱(Claude JSON)과 비즈니스 로직(quality/security/mx)을 분리하고, 코어 명령의 JSON 출력 계약을 문서화한다. 외부 동작 변화 없음 — 순수 준비 단계. 이미 코어가 잘 분리된 패키지 구조라 리스크가 낮다.
M2 — 표준 자산 이중 배포 (가장 값싼 승리)
① AGENTS.md 생성기 추가, CLAUDE.md를 @AGENTS.md 임포트 구조로 전환. ② 스킬 정본을 .agents/skills/로 옮기고 .claude/skills/는 설치 시 링크/복사. 이 단계만 끝나도 Codex·Gemini 사용자가 moai 스킬과 지침을 "읽을 수" 있게 된다.
M3 — 어댑터 레지스트리 + moai init --agent
spec-kit 패턴의 Go 인터페이스 레지스트리. 첫 타깃은 Codex(스킬 채택이 가장 성숙 + config.toml 단일 파일이라 어댑터가 얇음), 다음이 Gemini CLI(extension 매니페스트로 배포 단위까지 깔끔). 멀티 설치 안전성(디렉터리 격리, 매니페스트 분리)을 처음부터 설계.
M4 — moai mcp serve (수요 확인 후)
좁은 도구 세트(spec 조회, gate 실행, mx 스캔)로 시작, 읽기 전용 기본. CLI로 충분한 사용자는 건너뛰는 선택 계층.
M5 — 하네스별 UX 격차 보완
훅 부재 하네스를 위한 "게이트 리마인더"(스킬 지시문 강화), statusline 대체(프롬프트 내 상태 요약 명령 moai status --brief) 등. 완전 동등성이 아니라 "핵심 워크플로우의 신뢰 가능한 동작"이 목표.
스스로 확인해 볼 것 — M2가 끝난 시점에 Codex CLI를 설치하고 moai 프로젝트에서 $moai-plan이 뜨는지, moai gate가 셸에서 그대로 도는지 직접 확인해 보면 전략의 타당성을 가장 빨리 검증할 수 있다.

7리스크와 한계

  • 기능 하향 평준화 유혹: 모든 하네스에서 똑같이 만들려다 Claude Code의 강점(훅, 서브에이전트, output styles)을 깎아내리면 본말전도다. 원칙은 점진적 향상 — 코어는 어디서나, 최상 경험은 가장 능력 있는 하네스에서.
  • 테스트 매트릭스 확대: 하네스 3종 × 기능 조합의 회귀 비용. spec-kit처럼 어댑터를 자기완결 패키지로 격리하고 어댑터별 스모크 테스트를 CI에 두는 것이 완화책.
  • 표준의 유동성: Agent Skills는 표준화 1년 미만이고 벤더 확장(openai.yaml 등)이 계속 늘고 있다. 정본은 표준 코어(frontmatter + 본문)만 쓰고 벤더 확장은 어댑터 오버레이로 두면 표준 변화에 격리된다.
  • 지침 계층 압축 손실: .claude/rules/의 경로 조건부 로딩 같은 정교한 계층은 AGENTS.md 단일 파일로 옮기면 뭉개진다(32KiB 제한 포함). AGENTS.md에는 핵심 계약만 싣고 상세는 스킬 참조 파일로 내리는 배분 설계가 필요하다.
  • 훅 의존 안전장치: 현재 품질·보안 게이트 일부는 훅이 자동 실행을 보장한다. 훅 없는 하네스에서는 "에이전트가 지시를 따라야만 실행되는" 약한 보장으로 강등된다 — git pre-commit/pre-push 훅(하네스 무관)으로 최저선을 이중화하는 것을 권장.

FAQ

Q. MCP만 만들면 되는 것 아닌가요? 제일 유명한 표준인데요.
MCP는 "도구 호출" 하나만 표준화한다. 지침(에이전트 행동 규칙)과 스킬(작업 절차)은 MCP로 전달되지 않는다. 그리고 터미널 에이전트는 이미 셸을 갖고 있어서 CLI 호출이 MCP보다 마찰이 적다. MCP는 3대 표준 중 한 조각이며, moai-adk에는 후순위 선택 계층(M4)이 적절하다.
Q. 바이너리를 하네스별로 따로 만들어야 하나요?
아니다. 단일 moai 바이너리에 어댑터 레지스트리를 내장하는 구조(spec-kit과 동일)가 배포·버전 관리 면에서 압도적으로 낫다. moai init --agent codex처럼 초기화 시점에 어댑터를 선택하고, 복수 하네스 공존 설치를 허용하면 된다.
Q. LLM이 다르면 (GPT vs Claude vs Gemini) 스킬 품질이 달라지지 않나요?
달라진다. SKILL.md는 자연어 지시문이라 모델의 지시 이행 성향에 따라 편차가 생긴다. 완화책은 ① 스킬 지시를 검증 가능한 CLI 호출 중심으로 쓰기(모델 재량 최소화) ② 종료 코드/JSON으로 기계 검증 가능한 계약 만들기 ③ 하네스별 스모크 테스트. "지시는 짧게, 검증은 기계로"가 원칙이다.
Q. 지금 쓰는 Claude Code 환경이 이 작업 때문에 나빠지지 않나요?
M1·M2는 Claude Code 관점에서 동작 동일(CLAUDE.md가 임포트 한 줄로 바뀌는 것뿐, 로딩 결과는 같음)이다. 어댑터 레지스트리(M3)도 claude 어댑터가 현재 코드를 그대로 감싸는 형태라 기존 경로는 보존된다.
